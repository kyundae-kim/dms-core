# 테스트 정의서

## 1. 문서 목적

본 문서는 DMS SDK의 테스트 목표, 범위, 분류, 실행 기준, 환경 조건을 정의합니다.
이 문서는 다음에 답합니다.
- 어떤 수준의 테스트를 수행해야 하는가
- 어떤 기능을 자동 검증해야 하는가
- 통합 테스트는 어떤 조건에서 수행하거나 건너뛰는가
- 출시 판단에 필요한 검증 기준은 무엇인가

관련 문서:
- `docs/prd.md`
- `docs/srs.md`
- `docs/api.md`
- `docs/examples.md`
- `docs/config.md`

## 2. 테스트 목표

DMS SDK 테스트는 다음 목표를 가집니다.

- 문서 등록, 조회, 삭제의 핵심 생명주기를 지속적으로 검증한다.
- 문서 본문 저장소와 문서 정보 저장소의 정합성 규칙을 검증한다.
- 환경 기반 초기화와 명시적 의존성 주입 두 가지 조립 방식을 검증한다.
- 상태 점검, 로그, 오류 매핑 등 운영 관점 동작을 검증한다.
- 준비되지 않은 외부 환경 때문에 전체 검증이 불필요하게 실패하지 않도록 한다.

## 3. 테스트 범위

### 3.1 포함 범위

다음 항목은 자동 테스트 범위에 포함됩니다.

- SDK 생성
  - 환경 기반 조립
  - 명시적 의존성 주입
- 문서 업로드
- 문서 메타데이터 조회
- 문서 본문 전체 조회
- 문서 본문 스트리밍 조회
- 논리 삭제
- 완전 삭제
- 상태 점검
- 오류 매핑
- 구조화 로그
- 문서 정보 저장소 어댑터
- 문서 본문 저장소 어댑터
- 실제 외부 서비스와의 통합 경로
- `.env.example` 설정 예시의 필수 항목 포함 여부

### 3.2 제외 범위

현재 테스트 범위에는 다음이 포함되지 않습니다.

- 인증 기능
- 외부 공유용 임시 접근 링크
- 문서 검색 및 필터링
- 문서 버전 관리
- 비동기 후처리 연계
- 멀티테넌시
- 권한 모델 자체 구현

## 4. 테스트 수준 정의

### 4.1 단위 테스트

목적:
- 핵심 SDK 동작을 빠르게 검증한다.
- 외부 저장소 없이도 비즈니스 규칙과 오류 매핑을 확인한다.

특징:
- 가짜 메타데이터 저장소와 가짜 문서 본문 저장소를 사용한다.
- 실패 상황을 인위적으로 재현하기 쉽다.

주요 검증 대상:
- 업로드 성공 경로
- 스트리밍 조회 동작
- 청크 크기 검증
- 중복 문서 식별자 처리
- 파일명 정규화 및 저장 경로 생성
- 메타데이터 저장 실패 시 정리 동작
- 논리 삭제/완전 삭제 상태 처리
- 저장소 실패 시 상태 전이
- 메타데이터/본문 불일치 시 일관성 오류
- 문서 없음 오류
- 메타데이터 저장소 오류 매핑
- 상태 점검 결과 집계
- 종료 콜백 호출
- 구조화 로그 기록

관련 테스트 파일:
- `test_dms/test_sdk_behavior.py`

### 4.2 어댑터 테스트

목적:
- 저장소 어댑터와 환경 기반 조립 로직을 독립적으로 검증한다.

특징:
- 경량 DB 또는 가짜 클라이언트를 사용한다.
- 외부 서비스 없이도 어댑터 계약을 반복 검증할 수 있다.

주요 검증 대상:
- 문서 정보 저장소 round-trip
- 문서 정보 테이블 인덱스 생성
- 문서 본문 저장소 round-trip
- 문서 본문 스트리밍 round-trip
- PostgreSQL + MinIO 조합으로 SDK 생성
- SQLite + MinIO 조합으로 SDK 생성
- 공개 진입점 `create_sdk(env, ...)` 동작
- 시작 단계 상태 점검 실패 처리
- `.env.example` 필수 설정 포함 여부

관련 테스트 파일:
- `test_dms/test_infrastructure_adapters.py`

### 4.3 통합 테스트

목적:
- 실제 저장소 및 실제 환경 변수 조합으로 공개 SDK 흐름을 검증한다.

특징:
- 외부에 이미 준비된 PostgreSQL과 MinIO를 재사용한다.
- 서비스가 준비되지 않았으면 실패 대신 건너뛴다.
- 별도 테스트 전용 환경 변수 대신 실제 런타임 환경 변수 이름을 사용한다.

주요 검증 대상:
- 실제 PostgreSQL 메타데이터 저장소 동작
- 실제 MinIO 문서 본문 저장소 동작
- 실제 환경 변수 기반 SDK 생성
- 실제 업로드/조회/삭제 흐름
- 실제 상태 점검 흐름
- 실제 논리 삭제/완전 삭제 흐름

관련 테스트 파일:
- `test_dms/test_integration_adapters.py`

## 5. 테스트 시나리오 정의

### 5.1 SDK 생성

검증 항목:
- 환경 변수 매핑만으로 SDK를 생성할 수 있어야 한다.
- 명시적 의존성 주입으로 SDK를 생성할 수 있어야 한다.
- 환경 기반 방식과 명시적 의존성 주입 방식을 함께 전달하면 안 된다.
- 공통 설정 체계를 읽을 수 없거나 설정 로드가 실패하면 설정 오류가 발생해야 한다.

### 5.2 문서 업로드

검증 항목:
- 문서 본문과 메타데이터가 함께 저장되어야 한다.
- 호출자가 식별자를 생략하면 SDK가 새 식별자를 생성해야 한다.
- 중복 식별자는 거부되어야 한다.
- 파일명 정규화 규칙이 적용되어야 한다.
- 저장 경로는 `documents/` 접두사를 따라야 한다.
- 메타데이터 저장 실패 시 문서 본문 정리를 시도해야 한다.
- 메타데이터 저장과 정리 모두 실패하면 일관성 오류가 발생해야 한다.

### 5.3 문서 조회

검증 항목:
- 메타데이터 조회가 가능해야 한다.
- 문서 본문 전체 조회가 가능해야 한다.
- 스트리밍 조회가 가능해야 한다.
- 스트리밍 청크 크기 검증이 동작해야 한다.
- 메타데이터는 있지만 본문이 없으면 일관성 오류가 발생해야 한다.
- 존재하지 않는 문서는 문서 없음 오류로 구분되어야 한다.

### 5.4 문서 삭제

검증 항목:
- 논리 삭제 시 메타데이터 상태가 삭제 완료 상태로 남아야 한다.
- 완전 삭제 시 메타데이터가 제거되어야 한다.
- 삭제 시작 시 상태를 삭제 진행 상태로 전환해야 한다.
- 문서 본문 삭제 실패 시 실패 상태 전환을 시도해야 한다.
- 후속 메타데이터 처리 실패 시 삭제 진행 상태가 남을 수 있어야 한다.

### 5.5 상태 점검

검증 항목:
- 상태 점검 결과에 전체 성공 여부와 서비스별 결과가 포함되어야 한다.
- 실패한 서비스의 오류 정보가 노출되어야 한다.
- 환경 기반 생성 시 상태 점검이 활성화되어 있으면 시작 단계에서 검증해야 한다.
- 시작 단계 상태 점검 실패는 상태 점검 실패 오류가 되어야 한다.

### 5.6 로그와 운영 진단

검증 항목:
- 성공 경로에서 구조화 로그가 기록되어야 한다.
- 실패 경로에서 구조화 로그가 기록되어야 한다.
- 운영자가 문제 원인을 파악할 수 있는 수준의 진단 정보가 남아야 한다.

### 5.7 설정 및 예시 문서

검증 항목:
- `.env.example`에는 필수 설정 항목이 포함되어야 한다.
- 테스트가 기대하는 환경 변수 이름과 운영 환경 변수 이름이 일치해야 한다.

## 6. 테스트 데이터 및 대역

### 6.1 단위 테스트용 대역

단위 테스트에서는 다음 대역을 사용합니다.

- In-memory 메타데이터 저장소
- In-memory 문서 본문 저장소
- 실패를 강제하는 저장소 구현
- 건강/비건강 상태 점검 대역
- 종료 콜백 기록 대역

### 6.2 테스트 데이터 특성

대표 테스트 데이터는 다음을 포함합니다.

- 짧은 텍스트 파일
- 중복 문서 식별자
- 정규화가 필요한 파일명
- 존재하지 않는 문서 식별자
- 실패를 유도하는 저장소 대역
- 실제 저장소에 저장 후 삭제 가능한 임시 문서 식별자

## 7. 통합 테스트 실행 기준

### 7.1 필수 환경 변수

실제 통합 테스트는 다음 환경 변수가 준비되어 있을 때 수행합니다.

- `POSTGRES_DSN`
- `MINIO_ENDPOINT`
- `MINIO_ACCESS_KEY`
- `MINIO_SECRET_KEY`
- `MINIO_BUCKET`

### 7.2 선택 환경 변수

필요 시 다음 환경 변수를 함께 사용할 수 있습니다.

- `MINIO_SECURE`
- `DOCMESH_ENV`
- `DOCMESH_HEALTHCHECK_ENABLED`

### 7.3 건너뛰기 규칙

다음 경우 통합 테스트는 건너뜁니다.

- 필수 환경 변수가 하나라도 없는 경우
- PostgreSQL 드라이버가 설치되어 있지 않은 경우
- 외부 서비스 재사용이 불가능한 경우

원칙:
- 준비되지 않은 환경 때문에 전체 검증이 실패 상태로 오염되지 않아야 한다.
- 외부 서비스를 자동으로 새로 만들지 않고, 이미 준비된 서비스를 재사용한다.

## 8. 테스트 실행 방법

### 8.1 전체 테스트 실행

```bash
python3 -m pytest test_dms -q
```

### 8.2 파일 단위 실행 예시

```bash
python3 -m pytest test_dms/test_sdk_behavior.py -q
python3 -m pytest test_dms/test_infrastructure_adapters.py -q
python3 -m pytest test_dms/test_integration_adapters.py -q
```

### 8.3 기대 결과

- 단위 테스트와 어댑터 테스트는 로컬에서 반복 가능해야 한다.
- 통합 테스트는 외부 환경 준비 시에만 실제 검증을 수행해야 한다.
- 외부 환경 미준비 시 통합 테스트는 실패 대신 skip 되어야 한다.

## 9. 승인 기준

다음을 만족하면 테스트 기준을 충족한 것으로 봅니다.

- 핵심 문서 생명주기 테스트가 자동화되어 있다.
- 상태 점검 경로가 자동화되어 있다.
- 저장소 어댑터 round-trip 테스트가 존재한다.
- 환경 기반 조립과 명시적 의존성 주입 경로가 모두 검증된다.
- 실제 외부 서비스 재사용 통합 테스트가 준비되어 있다.
- 외부 환경 미준비 시 skip 정책이 적용된다.
- `.env.example` 필수 항목 검증이 포함된다.

## 10. 현재 확인 결과

이번 문서 작성 과정에서 코드와 테스트 파일을 기준으로 테스트 범위를 정리했습니다.
확인한 파일:
- `test_dms/test_sdk_behavior.py`
- `test_dms/test_infrastructure_adapters.py`
- `test_dms/test_integration_adapters.py`
- `.env.example`

실행 확인:
- `python3 -m pytest test_dms -q` 실행을 시도했으나 현재 환경에는 `pytest` 모듈이 없어 실행되지 않았습니다.
- 따라서 본 문서는 실제 테스트 코드 구조를 기준으로 작성되었고, 전체 테스트 통과 여부는 이번 턴에서 확인하지 못했습니다.
