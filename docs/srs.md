# 소프트웨어 요구사항 정의서

## 1. 목적 및 적용 범위

본 문서는 문서 관리 공통 구성요소가 충족해야 하는 소프트웨어 요구사항을 정의한다. 본 제품은 다른 업무 시스템에 통합되어 사용하며, 독립 실행형 API 서버를 제공하지 않는다.

## 2. 용어

- 문서 본문: 파일의 실제 바이트 데이터와 관련 속성이다.
- 문서 정보: 문서 식별자, 원본 파일명, 콘텐츠 유형, 크기, 저장 위치, 상태, 생성·수정 시각 및 부가 정보이다.
- 논리 삭제: 문서 정보를 보존한 채 문서를 삭제된 상태로 표시하는 처리이다.
- 완전 삭제: 문서 본문과 문서 정보를 모두 제거하는 처리이다.
- 불일치 상태: 문서 본문과 문서 정보의 존재 여부 또는 문서 상태가 서로 맞지 않는 상태이다.
- 커서 페이지네이션: 정렬 기준상의 마지막 위치를 불투명한 값으로 전달하여 다음 결과 묶음을 조회하는 방식이다.
- 비동기 스트림: 호출 흐름을 차단하지 않고 비동기 반복과 비동기 자원 정리를 지원하는 문서 본문 데이터 흐름이다.

## 3. 기능 요구사항

### 3.1 환경 설정 및 초기화

- SRS-CFG-001: 제품은 환경 기반 설정과 명시적 구성요소 조립 방식을 모두 제공해야 한다.
- SRS-CFG-002: 환경 기반 설정은 문서 정보 저장소와 문서 본문 저장소의 필수 설정을 확인해야 한다.
- SRS-CFG-003: 제품은 문서 정보 저장소로 PostgreSQL 또는 SQLite를 지원하고, 문서 본문 저장소로 MinIO를 지원해야 한다.
- SRS-CFG-004: 두 문서 정보 저장소 설정이 동시에 존재하는 자동 선택 상황에서는 우선 선택 결과와 주의 사항을 제공해야 한다. 엄격 설정 모드에서는 해당 모호성을 오류로 처리해야 한다.
- SRS-CFG-005: 설정 진단은 외부 서비스에 연결하거나 데이터를 변경하지 않고, 선택된 저장소 구성, 누락된 필수 값, 주의 사항 및 유효성 결과를 제공해야 한다.
- SRS-CFG-006: 설정 오류, 의존 요소 상태 확인 실패 및 저장소 연결 실패는 호출자가 원인을 구분할 수 있는 오류로 제공해야 한다.
- SRS-CFG-007: 환경 기반 조립은 기본적으로 시작 시 구성된 의존 요소의 상태를 확인해야 하며, 지정된 설정값으로 이 확인을 비활성화할 수 있어야 한다.
- SRS-CFG-008: 지원하지 않는 과거 PostgreSQL 연결 문자열 설정은 개별 필드 누락과 구분되는 금지 설정으로 진단하고 조립 전에 거부해야 한다.
- SRS-CFG-009: 설정 진단은 구조화된 결과와 비밀값을 포함하지 않는 운영자용 문자열 표현을 제공해야 하며, 설정 오류는 해당 진단 결과를 보존해야 한다.
- SRS-CFG-010: 시작 상태 확인 실패는 실패한 의존 요소, 실패 원인 및 재시도 가능 여부를 호출자가 구조적으로 확인할 수 있어야 한다.
- SRS-CFG-011: 제품은 공통 설정 모듈에서 검증·로드된 서비스 설정 묶음을 받아 환경변수에 접근하지 않고 SDK를 조립할 수 있어야 한다.
- SRS-CFG-012: 명시적 서비스 설정 묶음은 PostgreSQL과 SQLite 중 정확히 하나와 버킷이 지정된 MinIO 설정을 포함해야 한다.
- SRS-CFG-013: 명시적 서비스 설정 묶음 기반 조립은 환경 기반 조립과 동일한 오류 구분, 선택적 시작 상태 확인 및 안전한 종료 책임을 제공해야 한다.
- SRS-CFG-014: 서비스 설정 묶음 기반 조립이 일부 의존 요소를 생성한 뒤 실패하면 이미 생성한 자원을 최선의 방식으로 정리하고 최초 실패 원인을 보존해야 한다.
- SRS-CFG-015: 서비스 선택, 설정 진단 및 실제 조립은 동일한 형식화된 선택 정책에서 파생되어 서로 다른 저장소를 선택하지 않아야 한다.
- SRS-CFG-016: 별도 설정 집합을 진단하기 위한 일시적 프로세스 환경 적용은 진단 호환 경계에만 한정하고, 실제 서비스 조립에는 사용하지 않아야 한다. 해당 진단 중 동일한 환경 설정을 변경하는 외부 실행 흐름과의 동시 실행은 지원하지 않음을 안내해야 한다.
- SRS-CFG-017: 명시적 서비스 설정 묶음 기반 조립은 공통 실행 보안 정책 및 문서 본문 저장소 연결 보안 조건을 검증하고, 위반 시 일반 설정 오류와 구분 가능한 설정 오류로 거부해야 한다.

### 3.2 문서 등록

- SRS-UPL-001: 제품은 문서 본문, 파일명, 콘텐츠 유형, 선택적 문서 식별자, 부가 정보 및 작성자 정보를 받아 문서를 등록해야 한다.
- SRS-UPL-002: 제품은 메모리의 바이트 데이터와 선언된 크기를 가진 읽기 스트림을 통한 등록을 지원해야 한다.
- SRS-UPL-003: 스트림 등록 시 실제 읽은 크기가 선언된 크기와 일치하는지 검증해야 한다.
- SRS-UPL-004: 체크섬이 제공되면 등록 대상의 SHA-256 체크섬과 일치하는지 검증해야 한다.
- SRS-UPL-005: 제품은 유효하지 않은 파일명, 콘텐츠 유형, 크기, 스트림 단위 또는 부가 정보를 저장 전에 거부해야 한다.
- SRS-UPL-006: 최대 파일 크기 제한이 설정된 경우 모든 등록 방식에 동일하게 적용해야 한다.
- SRS-UPL-007: 제품은 중복 문서 식별자로 등록하려는 요청을 구분 가능한 중복 오류로 처리해야 한다.
- SRS-UPL-008: 본문 저장 후 문서 정보 저장에 실패하면 저장된 본문을 정리해야 한다. 본문 정리에도 실패하면 불일치 상태를 나타내는 오류를 제공해야 한다.
- SRS-UPL-009: 멱등성 키가 제공되면 동일한 범위와 동일한 요청에 대한 완료 결과를 재사용하고, 진행 중인 요청은 재시도 가능한 상태로 구분해야 한다. 스트림 등록의 멱등성 요청에는 체크섬을 요구해야 한다.
- SRS-UPL-010: 바이트 및 스트림 등록 요청은 명시적 멱등성 범위를 지원해야 한다. 멱등성 키와 함께 명시적 범위를 생략하는 방식은 폐기 예정(deprecated)이며, 기존 호출 호환성을 위해 범위가 없으면 작성자 또는 익명 범위를 사용하되 폐기 예정 경고를 제공해야 한다.
- SRS-UPL-011: 크기를 알 수 없는 스트림은 필수 양수 최대 크기로 제한된 임시 spool에 먼저 복사하여 크기와 SHA-256을 계산하고, 기존 엄격 스트림 등록 계약으로 처리해야 한다. 임시 자원은 성공·실패 시 모두 정리해야 한다.
- SRS-UPL-012: 크기를 알 수 없는 스트림의 읽기 단위는 양수이고 정해진 최대치를 초과하지 않아야 하며, SDK는 호출자가 제공한 입력 스트림을 닫지 않아야 한다.
- SRS-UPL-013: 바이트, 크기를 아는 스트림 및 크기를 알 수 없는 스트림 등록 진입점은 파일명, 콘텐츠 유형, 문서 식별자, 작성자, 부가 정보, 멱등성 값 및 크기 제한처럼 공통으로 받는 입력에 동일한 정규화와 검증 규칙을 적용하고, 저장 작업 전에 동일한 유효성 오류로 거부해야 한다.
- SRS-UPL-014: 제품은 비동기 스트림을 통한 문서 등록을 지원해야 하며, 크기를 아는 스트림과 크기를 알 수 없는 스트림에 동기 등록과 동일한 크기 제한, 체크섬, 멱등성, 정규화 및 오류 구분 규칙을 적용해야 한다.
- SRS-UPL-015: 비동기 스트림 등록은 호출자가 제공한 입력 스트림을 닫지 않아야 하며, 내부 임시 자원은 성공, 실패 또는 취소 시 모두 비동기적으로 정리해야 한다.

### 3.3 문서 조회

- SRS-GET-001: 제품은 문서 식별자로 삭제되지 않은 문서 정보를 조회해야 한다. 논리 삭제 또는 삭제 진행 상태인 문서는 일반 공개 조회에서 문서 없음으로 처리해야 한다.
- SRS-GET-002: 제품은 문서 식별자로 문서 본문을 조회해야 하며, 메모리 반환과 스트림 반환을 지원해야 한다.
- SRS-GET-003: 문서 본문 스트림은 반복 단위로 읽을 수 있고, 사용 종료 시 관련 자원을 해제할 수 있어야 한다.
- SRS-GET-004: 제품은 문서 상태를 기준으로 문서 정보를 목록 조회할 수 있어야 하며, 일반 공개 목록 조회는 삭제 완료 및 삭제 진행 상태의 문서를 기본적으로 제외해야 한다. 기존 오프셋/제한값 조회는 폐기 예정 호환 방식으로만 유지하고, 음수 오프셋과 0 이하 제한값을 거부해야 한다.
- SRS-GET-007: 제품의 기본 공개 목록 조회는 생성 시각과 문서 식별자의 내림차순 복합 순서를 사용하는 불투명 커서 페이지네이션이어야 한다. 커서는 상태 필터와 페이지 크기에 결합되어야 하며 결과는 항목, 다음 커서 및 추가 결과 여부를 포함해야 한다. 첫 조회는 커서 없이 시작하고, 마지막 페이지의 다음 커서는 없어야 한다.
- SRS-GET-005: 존재하지 않는 문서 정보 조회는 문서 없음 오류로 처리해야 한다.
- SRS-GET-006: 문서 정보는 존재하나 본문을 읽을 수 없으면 불일치 오류로 처리해야 한다.
- SRS-GET-008: 제품은 정확한 멱등성 범위와 키로 업로드 작업의 구조화된 상태를 조회해야 한다. 작업이 없으면 공개 작업 없음 오류를 제공해야 한다.
- SRS-GET-009: 일반 등록 결과, 문서 정보 조회 및 목록 결과는 내부 저장 위치를 포함하지 않는 공개용 문서 정보를 기본으로 반환해야 한다.
- SRS-GET-010: 내부 저장 위치를 포함하는 문서 정보 접근은 복구·관리 용도의 명시적 접근 경로로 분리해야 한다.
- SRS-GET-011: 논리 삭제 또는 삭제 진행 상태인 문서의 본문 및 본문 스트림 조회는 저장소 불일치가 아닌 삭제 상태 오류로 거부해야 한다. 삭제된 문서 정보와 삭제 상태 확인은 일반 공개 조회와 분리된 명시적 관리·복구 조회에서만 허용해야 한다.
- SRS-GET-012: 일반 공개 단건 조회, 오프셋 목록 조회 및 커서 목록 조회는 삭제 문서 은닉 정책을 일관되게 적용해야 하며, 호출자가 일반 조회의 선택 인자로 이를 우회할 수 없어야 한다.
- SRS-GET-013: 제품은 문서 본문을 비동기 반복 가능한 스트림으로 조회하는 인터페이스를 제공해야 하며, 전체 본문을 메모리에 적재하도록 강제하지 않아야 한다.
- SRS-GET-014: 비동기 문서 본문 스트림은 비동기 컨텍스트 관리와 명시적 비동기 종료를 지원하고, 정상 종료, 예외, 취소 및 일부 읽기 후 종료에서 관련 자원을 해제해야 한다.
- SRS-GET-015: 커서 페이지 크기는 양수인 허용 범위로 제한해야 하며, 변조되었거나 현재 조회 조건과 일치하지 않는 커서는 구분 가능한 유효성 오류로 거부해야 한다.

### 3.4 문서 삭제

- SRS-DEL-001: 제품은 논리 삭제와 완전 삭제를 지원해야 한다.
- SRS-DEL-002: 삭제는 본문 삭제와 문서 정보 상태 변경 또는 제거를 일관된 순서로 처리해야 한다.
- SRS-DEL-003: 삭제 중 본문 삭제에 실패하면 문서 정보에 실패 상태를 남겨 불완전한 상태를 식별할 수 있어야 한다.
- SRS-DEL-004: 본문 삭제 후 문서 정보 처리에 실패하면 삭제 진행 상태를 남겨 복구 대상임을 식별할 수 있어야 한다.
- SRS-DEL-005: 논리 삭제와 완전 삭제는 각각 명시적 공개 메서드로 제공해야 한다. 기존 통합 삭제 메서드와 완전 삭제 선택 인자는 폐기 예정(deprecated) 호환 API로 유지하되, 호출자에게 논리 삭제 또는 완전 삭제의 명시적 공개 메서드 사용을 안내해야 한다.
- SRS-DEL-006: 논리 삭제와 완전 삭제의 공개 결과는 문서 식별자, 삭제 수행 여부, 완전 삭제 여부 및 최종 상태를 포함하는 공식 결과 모델로 반환해야 하며, 해당 모델은 명시적으로 정의된 직렬화 결과를 제공해야 한다.

### 3.5 상태 점검 및 복구

- SRS-REC-001: 제품은 문서 정보의 존재 여부, 문서 본문의 존재 여부, 문서 상태 및 일관성 판단 결과를 제공해야 한다.
- SRS-REC-002: 문서 정보가 없는 점검 요청은 문서 없음 오류가 아닌 점검 결과로 제공해야 한다.
- SRS-REC-003: 제품은 삭제 진행 상태에서 본문이 이미 제거된 문서의 삭제 완료 처리를 지원해야 한다.
- SRS-REC-004: 제품은 문서 정보는 존재하지만 본문이 없는 문서를 실패 상태로 표시하는 복구를 지원해야 한다.
- SRS-REC-005: 제품은 문서 정보 없이 남은 본문을 명시적으로 제공된 저장 위치를 기준으로 제거하는 복구를 지원해야 한다.
- SRS-REC-006: 복구는 실제 변경 없이 결과를 미리 확인하는 실행 방식을 지원해야 한다.
- SRS-REC-007: 제품은 실패 또는 삭제 진행 상태의 문서를 제한된 범위로 조회하고, 각 복구 항목의 성공 또는 실패 결과를 개별적으로 제공해야 한다.
- SRS-REC-008: 일괄 복구 결과는 조회·적격·적용·건너뜀·실패 수를 안정적으로 제공하고, 오류 처리 중 재점검까지 실패해도 나머지 항목 처리를 중단하지 않아야 한다.

### 3.6 상태 확인 및 종료

- SRS-OPS-001: 제품은 구성된 문서 정보 저장소와 문서 본문 저장소의 상태를 개별 결과로 제공해야 한다.
- SRS-OPS-002: 상태 확인 결과에는 확인 시각을 포함해야 하며, 각 대상 결과에는 대상 이름, 정상 여부 및 실패 원인을 포함해야 한다.
- SRS-OPS-003: 제품은 사용 종료 시 저장소 연결과 등록된 정리 자원을 안전하게 해제해야 하며, 반복 종료 호출이 오류를 일으키지 않아야 한다.
- SRS-OPS-004: 바이트 등록, 문서 정보·본문 조회, 오프셋 목록 조회, 삭제 및 상태 확인은 처리 결과와 관련 문서 식별자 또는 목록 조건을 포함한 구조화된 운영 로그를 남겨야 한다.
- SRS-OPS-005: 환경 또는 서비스 설정으로 생성한 SDK는 자신이 생성한 서비스 묶음의 종료 책임을 가져야 하며, 조립 실패 시 생성된 자원을 정리해야 한다.
- SRS-OPS-006: SDK는 동기 컨텍스트 관리자로 사용할 수 있어야 하며 컨텍스트 종료 시 반복 호출에 안전한 종료 동작을 수행해야 한다. 문서 본문 스트림도 동기 컨텍스트 관리자로 사용할 수 있어야 하며 정상 종료와 예외 종료 모두에서 관련 자원을 해제해야 한다.
- SRS-OPS-007: 비동기 스트리밍 인터페이스를 제공하는 SDK는 비동기 컨텍스트 관리와 비동기 종료를 지원해야 하며, 반복 종료와 취소 상황에서도 자신이 생성한 자원을 안전하게 정리해야 한다.

## 4. 데이터 및 상태 요구사항

- SRS-DAT-001: 문서 정보는 문서 식별자, 원본 파일명, 콘텐츠 유형, 파일 크기, 저장 위치, 상태, 생성·수정 시각을 보존해야 한다.
- SRS-DAT-002: 문서 정보는 선택적 체크섬, 삭제 시각, 작성자 및 부가 정보를 보존할 수 있어야 한다.
- SRS-DAT-003: 문서 상태는 사용 가능, 삭제 진행, 삭제 완료 및 실패 상태를 구분해야 한다.
- SRS-DAT-004: 정상 등록이 완료된 문서는 사용 가능 상태여야 한다.
- SRS-DAT-005: `UPLOADED` 상태는 과거 데이터 호환을 위한 폐기 예정(deprecated) 값이며 정상 등록 흐름에서 새로 생성하지 않아야 한다. 새 문서에는 `AVAILABLE` 상태를 사용해야 한다.
- SRS-DAT-006: 공개 문서 정보는 내부 저장 위치를 제외한 문서 식별자, 원본 파일명, 콘텐츠 유형, 파일 크기, 상태, 생성·수정 시각, 선택적 체크섬·삭제 시각·작성자 및 부가 정보를 포함하는 공식 모델이어야 한다.
- SRS-DAT-007: 공개 문서 정보와 삭제 결과의 직렬화는 열거 상태를 문자열 값으로, 날짜·시각을 시간대가 포함된 표준 문자열로, 중첩 부가 정보를 JSON 호환 값으로 변환해야 하며 내부 저장 위치나 직렬화할 수 없는 런타임 객체를 포함하지 않아야 한다.

## 5. 비기능 요구사항

- SRS-NFR-001: 공개 API는 호출자가 유효성 오류, 중복, 문서 없음, 삭제 상태, 객체 저장소 오류, 문서 정보 저장소 오류, 설정 오류, 상태 확인 실패, 멱등성 충돌·진행 상태 및 불일치를 구분할 수 있는 오류 유형을 제공해야 한다.
- SRS-NFR-002: 문서 처리와 복구는 부분 실패가 발생해도 원인과 후속 조치 판단에 필요한 상태를 보존해야 한다.
- SRS-NFR-003: 자동화된 검증은 등록, 조회, 삭제, 상태 점검, 복구, 설정 진단 및 상태 확인의 핵심 흐름을 포함해야 한다.
- SRS-NFR-004: 실제 저장소 연동 검증은 준비된 외부 서비스를 재사용해야 하며, 외부 서비스가 준비되지 않은 환경에서는 해당 검증을 명확히 구분해야 한다.
- SRS-NFR-005: 사용자 안내 문서, 제품 요구사항 및 본 문서는 제품의 실제 제공 범위와 일치해야 한다.
- SRS-NFR-006: 기본 공개 등록 및 조회 결과의 문서 정보는 내부 저장 위치를 포함하지 않아야 한다.
- SRS-NFR-007: 부가 정보 스키마 검증은 외부 모델 라이브러리 의존 없이 버전 및 필드별 오류를 제공하고 기존 validator 계약을 유지해야 한다.
- SRS-NFR-008: 내보낸 복구 계획 실행은 각 작업 직전에 현재 상태를 재점검해야 하며, 복구 감사 callback 실패는 복구 결과를 가리지 않아야 한다.
- SRS-NFR-009: 모든 공개 SDK 오류는 예외 유형과 별개로 안정된 기계 판독용 오류 코드, 상위 오류 분류 및 재시도 가능 여부를 제공해야 한다. 동일한 실패 의미는 진입점이나 하위 저장소 구현과 관계없이 같은 코드와 분류를 사용해야 하며, 새 오류 코드 추가는 기존 코드의 의미를 변경하지 않아야 한다.
- SRS-NFR-010: 제품은 독립 실행형 API 서버를 제공하지 않더라도, 호스트 시스템이 공개 오류 분류를 HTTP 응답으로 변환할 때 적용할 권장 매핑을 문서화해야 한다. 이 매핑은 전송 계층 권고이며 SDK 오류 자체에 HTTP 의존성을 추가하지 않아야 한다.

### 5.1 공개 오류의 권장 HTTP 매핑

| 공개 오류 의미 | 권장 HTTP 상태 | 적용 기준 |
| --- | --- | --- |
| 입력값 또는 커서 유효성 오류 | 400 Bad Request | 요청 형식, 값, 크기, 체크섬 또는 커서가 계약을 위반한 경우 |
| 문서 또는 작업 없음 | 404 Not Found | 공개 조회 대상이 없거나 삭제 은닉 정책에 따라 없음으로 처리한 경우 |
| 중복 또는 멱등성 충돌 | 409 Conflict | 식별자 중복, 동일 키의 다른 요청 또는 현재 상태와 충돌한 경우 |
| 삭제 상태 또는 처리할 수 없는 상태 | 409 Conflict | 대상 상태가 요청 작업과 충돌한 경우 |
| 크기 제한 초과 | 413 Content Too Large | 문서 본문이 허용된 최대 크기를 초과한 경우 |
| 진행 중인 멱등성 요청 | 425 Too Early | 같은 범위와 키의 작업이 아직 완료되지 않아 재시도가 필요한 경우 |
| 설정 오류 | 500 Internal Server Error | 서버 통합 또는 실행 설정이 잘못된 경우 |
| 문서 정보 저장소 또는 문서 본문 저장소 오류 | 503 Service Unavailable | 의존 저장소가 일시적으로 요청을 처리할 수 없는 경우 |
| 상태 확인 실패 | 503 Service Unavailable | 필수 의존 요소가 준비되지 않은 경우 |
| 저장소 불일치 또는 복구가 필요한 부분 실패 | 500 Internal Server Error | 자동으로 정상 응답을 보장할 수 없고 운영 점검이 필요한 경우 |

- SRS-NFR-011: 권장 HTTP 응답 본문은 안정된 오류 코드, 상위 오류 분류, 재시도 가능 여부 및 비밀값을 제외한 진단 메시지를 포함해야 한다.
- SRS-NFR-012: 호스트 시스템은 보안 정책에 따라 진단 메시지를 축약할 수 있지만, 같은 공개 오류 의미에 대한 HTTP 상태와 기계 판독용 오류 코드를 임의로 변경하지 않아야 한다.
