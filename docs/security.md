# 보안 가이드

## 1. 문서 목적

본 문서는 현재 DMS SDK의 실제 보안 경계와, SDK를 사용하는 애플리케이션이 지켜야 하는 운영 지침을 정의합니다.
코드가 강제하는 동작과 운영 권장 사항을 구분합니다.

관련 문서:
- `docs/api.md`
- `docs/config.md`
- `docs/examples.md`
- `docs/messaging.md`

## 2. 현재 SDK가 제공하는 보안 관련 동작

### 2.1 사용자 metadata 정책

기본 `DefaultMetadataPolicy`는 bytes 업로드와 스트리밍 업로드 전에 사용자 metadata를 검증하고 정규화합니다.

- 최상위 키는 문자열이어야 합니다.
- 값은 JSON 직렬화 가능해야 합니다.
- 기본 직렬화 크기 상한은 16,384 UTF-8 바이트입니다.
- 기본 중첩 깊이 상한은 8입니다.
- 다음 종류의 민감 키는 대소문자를 구분하지 않고 거부합니다: 비밀번호, secret, token, API key, authorization, credential, private key.
- 정규화 결과는 입력 객체와 분리된 JSON 호환 사본입니다.

검증 실패는 객체 저장 전에 `ValidationError`로 노출됩니다. 호출자가 `metadata_validator`를 제공하면 기본 정책 대신 그 검증기가 사용됩니다. 사용자 검증기도 입력을 변경하지 않고, 민감 정보가 포함되지 않도록 동등하거나 더 강한 정책을 적용해야 합니다.

이 정책은 최상위 키만 검사합니다. 민감 정보를 metadata에 넣지 않는 책임은 호출자에게 있으며, 키 이름을 바꾸거나 중첩 구조에 넣는 방식에 의존해서는 안 됩니다.

### 2.2 설정 진단의 secret 보호

`diagnose_environment(env)`는 연결이나 클라이언트 생성을 수행하지 않으며, 결과에 환경 변수 값이나 secret을 포함하지 않습니다.
진단 결과와 예외 메시지는 값이 아닌 누락 키·backend 선택·경고 중심으로 운영 로그에 남깁니다.

### 2.3 전송 보안 설정

MinIO 연결의 TLS 사용 여부는 `MINIO_SECURE`로 설정합니다. SDK는 이 값에 따라 클라이언트를 조립하지만, 운영 환경에서 TLS를 강제하지는 않습니다.

- 인터넷 또는 신뢰할 수 없는 네트워크를 경유하는 MinIO 연결에는 `MINIO_SECURE=true`와 신뢰 가능한 인증서 구성을 사용합니다.
- `MINIO_SECURE=false`는 로컬 개발 또는 별도의 전송 계층 보호가 검증된 환경으로 제한합니다.

### 2.4 무결성과 보안의 구분

업로드 checksum과 스트리밍 업로드의 SHA-256 검증은 전송·저장 과정의 무결성 확인 수단입니다. 이는 인증, 권한 부여, 암호화 또는 악성 파일 검사 기능을 제공하지 않습니다.

## 3. 현재 범위 밖의 보안 기능

현재 DMS SDK는 다음을 제공하지 않습니다.

- 사용자 인증 또는 접근 권한 부여
- 문서별 접근 제어, 멀티테넌시 격리
- 문서 본문 암호화 또는 키 관리
- 악성 코드 검사, 콘텐츠 유형 신뢰성 검사, DLP
- 감사 로그 저장
- 메시지 브로커 기반 보안 계약

애플리케이션은 SDK 호출 전 인증·인가를 수행하고, 문서 식별자에 대한 소유권 및 접근 권한을 확인해야 합니다. 저장소 접근 정책, bucket 권한, 암호화, 악성 파일 검사 및 감사 추적은 배포 환경 또는 상위 서비스의 책임입니다.

## 4. 자격 증명과 환경 변수 운영

- `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, PostgreSQL 연결 정보 등 자격 증명을 소스코드, 문서 예시, 테스트 fixture, 로그에 기록하지 않습니다.
- 배포 환경에서는 비밀 관리 시스템 또는 배포 플랫폼의 secret 주입 기능을 사용합니다.
- `.env.example`은 변수 이름과 비민감 예시만 제공하는 템플릿입니다. 실제 `.env` 파일은 버전 관리 대상에 포함하지 않습니다.
- MinIO와 PostgreSQL 계정은 필요한 bucket·database 작업에 한정된 최소 권한을 사용합니다.
- credential 교체 시에는 기존 환경 값을 폐기하고, `diagnose_environment()`로 값 자체를 노출하지 않고 설정 구조를 확인합니다.

## 5. 개발 확인 목록

배포 또는 통합 전에 다음을 확인합니다.

- [ ] 운영 MinIO 연결에서 `MINIO_SECURE=true`와 인증서 검증 정책을 적용했다.
- [ ] 자격 증명과 사용자 token을 `metadata`에 포함하지 않았다.
- [ ] 사용자 metadata 검증기를 교체했다면 기본 정책보다 약해지지 않도록 검토했다.
- [ ] 애플리케이션 계층에서 인증·인가와 문서 소유권 검사를 수행한다.
- [ ] MinIO bucket 및 PostgreSQL 계정에 최소 권한을 적용했다.
- [ ] checksum을 보안 경계가 아닌 무결성 점검으로만 사용한다.
